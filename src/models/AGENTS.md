# Model Agent Rules

These rules apply to files under `src/models`.

Follow the backend architecture and TypeScript rules defined in the root `AGENTS.md`. Model rules define the source of truth for domain data shapes used by controllers, services, repositories, DTOs, and utilities.

Before changing model code, read `<project-root>/.agents/skills/nodejs-backend-patterns/SKILL.md` for backend type boundary and maintainability patterns. Apply the model ownership and naming rules in this file over generic examples in that skill.

---

## Model Responsibility

Models own reusable domain shapes that are not generated by Prisma or inferred from Zod DTOs.

Define domain-level TypeScript utilities in the appropriate model file:

- interfaces and types for service-layer payloads and results
- enums and constants representing domain concepts
- view/response shapes that differ from the Prisma model
- nested object shapes shared across services or repositories
- pagination result wrappers

Do not define reusable domain types, interfaces, enums, or constants inside services, controllers, repositories, routes, or utilities when they belong to an existing domain model.

Avoid service-local domain type definitions. If a service needs a named payload, result, enum, or constant for a domain concept, define it in the matching model file and import it.

Small private helper types are allowed inside a service only when they are implementation-specific, not reused across files, and not a domain concept.

---

## Prisma-Generated Types

Prisma generates types from the schema. Prefer Prisma-generated types (`Post`, `User`, `Comment`, etc.) for persisted entity shapes rather than redefining them manually in model files.

Use Prisma utility types for partial shapes:

```ts
import type { Prisma } from '@prisma/client';

type PostWithAuthor = Prisma.PostGetPayload<{
    include: { author: true };
}>;

type PostCreateInput = Prisma.PostCreateInput;
```

Only define a custom model type when the Prisma-generated type is genuinely insufficient — for example, when a response shape omits sensitive fields, merges data from multiple tables, or represents a computed view.

Do not redefine a type that already exists in `@prisma/client`. Import it directly.

---

## Naming

Keep model names clean, explicit, and domain-oriented.

Use names that describe the domain concept directly:

- `Post`
- `PostStatus`
- `PostListItem`
- `PostCreatePayload`
- `PostUpdatePayload`
- `FeedCursor`

Avoid vague or implementation-oriented names:

- `Data`
- `Result`
- `Payload`
- `TempType`
- `PostType2`
- `ServicePost`

Use the existing naming style in nearby model files before introducing a new name.

## DTO Alignment

When a model type is intentionally used as the same contract as a DTO-inferred type, its TypeScript fields must match the DTO exactly:

- same property names
- same required/optional fields
- same nullable fields
- same array/object nesting
- same primitive types
- same enum/union values

Do not create a model type that is only "close enough" to the DTO shape.

When DTO schemas and model types share enum-like values, define the canonical values once in the model layer when practical, then import or mirror them in DTOs consistently. Do not create divergent string unions for the same domain concept.

If a DTO and a persisted/response model intentionally differ, keep the boundary explicit:

- DTO files validate request/response input shapes.
- Model files define domain and service-layer shapes.
- Services map between DTO and model shapes when the shapes are genuinely different.

Do not silently pass a mismatched DTO shape as a model shape.

---

## Enums

Prefer TypeScript string enums or `as const` objects over plain string union types for domain enums that are also represented as Prisma enums:

```ts
// Preferred when the value is also a Prisma enum
export const PostStatus = {
    DRAFT: 'DRAFT',
    PUBLISHED: 'PUBLISHED',
    ARCHIVED: 'ARCHIVED',
} as const;

export type PostStatus = (typeof PostStatus)[keyof typeof PostStatus];
```

Keep enum values in sync with the Prisma schema enum definition. Do not define a separate string union with different values for the same concept.

---

## Use the Correct Domain Model

Always import types from the model file for the domain being changed:

- Post service changes → `src/models/post.model.ts`
- User service changes → `src/models/user.model.ts`
- Comment service changes → `src/models/comment.model.ts`

Before adding or changing a type, search the matching model file first. Reuse existing model types when they already describe the shape.

Do not define a duplicate type in a service just because it is convenient for one method.

---

## Adding New Model Utilities

Do not add new model types, interfaces, enums, or constants unless the task requires them or the user explicitly asks for them.

When a new model utility is necessary:

- Put it in the closest matching domain model file.
- Export it only if it is used outside the file.
- Keep the shape minimal and specific to the use case.
- Prefer composing existing model types over duplicating fields.
- Keep constants close to the model concept they describe.

If no matching model file exists, create the smallest appropriate model file only when the task requires a new domain concept.

---

## Imports

Do not import types from `dist` or any build output directory; that is compiled output.

Import source model types from `src/models`.

Import Prisma-generated types from `@prisma/client`.

Use type-only imports for TypeScript-only model dependencies when practical:

```ts
import type { Post } from '@prisma/client';
import type { PostListItem } from '../models/post.model';
```

## Final Response Note

When changing model files, include a short model note in the final response:

```
Model note:
- Model file changed:
- Prisma-generated type reused: yes/no
- Existing model reused: yes/no
- DTO/model shape checked: yes/no
- New type/interface/enum/constant added: yes/no
```
